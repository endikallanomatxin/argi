String should follow the same ownership split as lists:

- `String` is the owning type.
- it should be backed by `Allocation`.
- borrowed string views should stay separate and non-owning.
- direct indexing on `String` itself should stay unavailable.
- byte-level or text-level indexing should go through explicit views/operations.

Terminology to keep stable:

- `bytes`: raw UTF-8 bytes
- `codepoints`: decoded Unicode scalar values
- `graphemes`: user-perceived text units

Proposed minimal direction:

```
String : Type = (
    .allocation : Allocation
    .length     : UIntNative
)

StringViewRO : Type = (
    .string : &String
    .start  : UIntNative
    .length : UIntNative
)

StringViewRW : Type = (
    .string : $&String
    .start  : UIntNative
    .length : UIntNative
)
```

The important design point is the same as for list views:

- copying a string view copies only the descriptor,
- not the backing bytes,
- and not the ownership.

Likely split:

- explicit byte access for the owning `String`,
- later, explicit view constructors:
  - `view_bytes(...)`
  - `view_codepoints(...)`
  - `view_graphemes(...)`
- and indexing should happen on those views rather than on `String` itself.
